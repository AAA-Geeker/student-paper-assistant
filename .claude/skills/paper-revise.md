# 论文修改 Skill

## 目标
接收导师或审稿人的反馈意见，逐条解析、定位、生成修改方案，并应用到论文中。

## 前置条件
- 已有论文内容（在 WorkBuddy 中）
- 已读取 [[advisor-feedback]]（了解导师反馈模式）
- 用户提供反馈文本

## 执行流程

### Step 1：解析反馈
将导师/审稿人的反馈拆解为具体的可操作条目：

对于每条反馈，标注：
- **类型**：结构 / 论证 / 写作 / 格式
- **影响范围**：单段 / 单节 / 全文
- **紧急程度**：必须改 / 建议改 / 可选

参考 [[advisor-feedback]] 中的导师话术翻译表（如"这里逻辑不对" = 因果关系需重新梳理）。

### Step 2：定位原文
对于每条反馈，找到论文中对应的位置（段落、章节）。
只读取相关段落的上下文（前后各 1-2 段），不需要传全文。

### Step 3：生成修改方案
对于每条反馈，生成 2-3 个修改方案：
- **方案 A**：最小改动（只改必要的地方）
- **方案 B**：中等改动（重写相关段落）
- **方案 C**：深度改动（调整相关章节结构）

每个方案标注：
- 改动范围
- 优缺点
- 推荐指数（⭐1-5）

### Step 4：用户确认
将修改方案呈现给用户，让用户选择或组合方案。
对于"必须改"级别的反馈，强烈建议选择至少方案 B。

### Step 5：应用修改 + 生成说明
将用户选择的修改应用到论文中，并生成一份**修改说明**：
```
修改项 #1：[反馈原文摘要]
  → 应用方案：B
  → 改动位置：第 2.3 节，第 3-5 段
  → 改动内容摘要：重写了方法对比的逻辑链，增加了与 SOTA 的对比分析
  → 导师原话："这里逻辑不对" → 原来是并列罗列方法，现改为"问题→现有解法→不足→我们的解法"的递进结构
```

## 输出到 WorkBuddy
1. 更新论文内容
2. 可选：在论文末尾添加 `<!-- 修改记录 -->` 注释

## 模型使用策略
- Step 1-2（解析+定位）：本地处理 + 简单 LLM 调用
- Step 3（生成方案）：GPT-4o-mini 即可
- Step 5（应用修改）：GPT-4o-mini 逐段修改
